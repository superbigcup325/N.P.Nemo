# N.P.Nemo - 项目规划文档

## 项目配置总览

| 维度 | 决策 |
|:---|:---|
| **游戏类型** | 回合制卡牌Roguelike（杀戮尖塔玩法） |
| **视觉方案** | 纯手搓SVG（零图片），渐进式（几何抽象→精细） |
| **前端** | React + Zustand + Framer Motion |
| **后端** | FastAPI + SQLite |
| **通信** | REST API |
| **逻辑权威** | 后端（卡牌结算、敌人AI、RNG均在后端） |
| **多人模式** | 纯单机 |
| **原型规模** | 最小可玩原型 |
| **遗物系统** | 原型不含 |
| **核心原则** | 高内聚低耦合，保证可扩展性 |

---

## 阶段性目标

### 阶段一：工程基建与架构搭建

**目标**：搭建前后端项目骨架，定义核心数据契约与SVG渲染基础设施，确保后续开发有清晰的模块边界。

**做法**：

- **前端**：使用 Vite 初始化 React + TypeScript 项目；建立分层目录结构（`components/`、`systems/`、`stores/`、`services/`、`svg/`）；封装 SVG 渲染基础设施（SVG 画布容器、坐标系统、图层分组容器 `layer-background / layer-gameobjects / layer-fx / layer-ui`）；配置 Zustand store 骨架（游戏状态、UI状态分离）；封装 API Service 层与后端通信；配置 Framer Motion 动画基础工具。
- **后端**：初始化 FastAPI 项目，建立分层结构（`routers/`、`services/`、`models/`、`schemas/`、`core/`）；定义核心 Pydantic Schema（GameState、Card、Enemy、Player 等数据契约）；设计游戏状态机框架（`MapState → BattleState → RewardState → MapState`）；配置 SQLite + SQLAlchemy；定义 REST API 路由骨架（`/game/start`、`/game/action`、`/game/state`）。
- **跨端**：编写 API 接口文档（OpenAPI），前后端共享数据契约定义；配置 CORS、开发代理。

**验收标准**：

1. 前后端项目均可独立启动，无报错
2. 前端能渲染一个空白 SVG 画布，包含四个图层分组容器
3. 前端调用 `/game/start` 能从后端获取初始 GameState 并在 Zustand store 中存储
4. 后端 SQLite 可正常读写，数据模型与 Schema 一致
5. 目录结构清晰，各模块职责边界明确，无循环依赖

### 阶段二：核心战斗系统（1v1原型）

**目标**：实现1v1卡牌战斗的完整最小循环——抽牌→出牌→结束回合→敌人行动→循环，直到一方HP归零。

**做法**：

- **后端战斗引擎**：实现回合制状态机（`PlayerTurn → EnemyTurn → PlayerTurn...`）；实现牌组管理器（抽牌堆、手牌、弃牌堆、消耗堆的流转逻辑）；实现卡牌效果执行器（基于效果类型的策略模式，如 `DealDamageEffect`、`GainBlockEffect`、`ApplyBuffEffect`）；实现能量系统（每回合重置，出牌扣费）；实现敌人意图系统（下回合预告：攻击X/防御/强化）；实现敌人AI（简单模式：随机选择意图）；实现战斗结算判定（HP≤0时触发胜负）。
- **前端战斗UI**：实现SVG卡牌组件（几何抽象风格：矩形卡体+文字+色块标识类型）；实现手牌区域（扇形排列，Framer Motion 实现悬浮放大/选中高亮）；实现战斗场景布局（玩家区域底部、敌人区域顶部、手牌区域底部前方）；实现敌人意图SVG图标显示；实现HP/能量/护甲的SVG数值显示；实现出牌交互（点击卡牌→选择目标→发送action到后端→接收新状态→动画过渡）；实现回合结束按钮。
- **卡牌内容**：设计5-8张基础卡牌（打击x4、防御x4、重击x1，类似杀戮尖塔铁甲战士初始牌组）。

**验收标准**：

1. 能开始一场1v1战斗，初始手牌5张，能量3点
2. 点击手牌中的攻击卡→点击敌人→敌人扣血（有攻击动画）→手牌消耗→能量扣除
3. 点击防御卡→玩家获得护甲（有防御动画）→手牌消耗→能量扣除
4. 点击"结束回合"→弃牌堆洗回抽牌堆→抽5张新牌→能量重置→敌人执行意图动作
5. 敌人意图在玩家回合清晰可见
6. 玩家HP或敌人HP归零时，战斗正确结束并显示胜负
7. 所有状态变更均由后端计算，前端仅渲染后端返回的状态

### 阶段三：地图推进与局外成长

**目标**：实现单局游戏的完整流程——从地图选择房间→进入战斗/事件→获得奖励→构建牌组→推进到下一层。

**做法**：

- **后端地图系统**：先实现**线性地图**（每层3-5个房间，按序排列，类型随机生成）；实现房间类型分配器（战斗60%、休息20%、商店10%、事件10%）；实现房间进入逻辑（根据房间类型切换对应状态）；实现战斗奖励生成器（金币、3选1卡牌、药水槽位）；实现休息站点逻辑（回复30%HP或升级一张卡牌）；实现商店系统（购买卡牌、移除卡牌、购买药水）；实现层间推进逻辑（3层，每层末尾有BOSS）；实现RNG种子管理（确保可复现）。
- **后端牌组构建**：实现卡牌添加/移除逻辑；实现卡牌升级逻辑（升级版卡牌效果增强）；实现初始牌组配置。
- **前端地图UI**：实现SVG线性地图渲染（节点+连线+当前位置标记）；实现房间类型SVG图标（剑=战斗、火=休息、金币=商店、?=事件）；实现房间选择交互（点击下一个节点进入）；实现奖励选择界面（3张卡牌SVG展示，点击选择）；实现休息站点界面（两个选项按钮）；实现商店界面（商品列表+金币显示+购买交互）。
- **前端牌组查看**：实现牌组浏览面板（查看当前所有卡牌）。

**验收标准**：

1. 游戏开始后显示线性地图，玩家从第一个节点出发
2. 点击下一个节点→进入对应房间类型→完成后返回地图→可继续前进
3. 战斗胜利后弹出奖励界面，选择卡牌后卡牌加入牌组
4. 休息站点可选择回血或升级卡牌，效果正确
5. 商店可购买/移除卡牌，金币正确扣除
6. 通过3层地图并击败最终BOSS后，显示胜利画面
7. 任意时刻HP归零，显示失败画面
8. 牌组浏览面板能正确显示当前所有卡牌

### 阶段四：内容扩展与体验打磨

**目标**：从1v1原型扩展到1v多战斗，增加卡牌/敌人种类，实现分支地图，优化视觉动画体验。

**做法**：

- **多敌人战斗**：后端扩展战斗引擎支持多敌人（敌人列表、目标选择、AOE效果）；前端实现多敌人布局与目标选择交互。
- **分支地图**：后端实现分支地图生成算法（每层3-4条路径，节点间有连线）；前端实现分支地图SVG渲染与路径选择交互。
- **内容扩展**：设计15-20张卡牌（覆盖攻击、技能、能力三种类型）；设计5-8种敌人（含2个BOSS），每种有独特AI模式；设计3-5个简单事件（选择→结果）。
- **视觉打磨**：将几何抽象卡牌逐步替换为精细SVG路径；增加战斗特效动画（攻击拖尾、护甲碎裂、伤害数字飘字）；增加地图过渡动画；增加卡牌抽/出/弃的流畅动画。
- **音效系统**：接入Web Audio API，为基础操作添加音效（出牌、攻击、受伤、按钮点击）。

**验收标准**：

1. 战斗中可同时面对2-3个敌人，目标选择正确，AOE卡牌影响所有敌人
2. 地图为分支结构，玩家可在分叉口选择不同路径
3. 卡牌种类覆盖攻击/技能/能力三大类，每种至少5张
4. 每种敌人有独特的意图模式和行为逻辑
5. 卡牌动画流畅（抽牌滑入、出牌飞向目标、弃牌滑出）
6. 战斗特效清晰可辨（攻击/防御/增益/减益各有视觉反馈）
7. 音效与操作同步，无卡顿
